# 04 — Recommender Specification (rev. 3, final)

> Working spec **and** seed for the mandatory `recommender-dokumentacija.md`. Course hard rules: implementation matches this document; every recorded signal is used in scoring; every recommendation is explainable; inputs are actually written by the running app.

## 1. Algorithm choice (final)

ML.NET's recommendation catalog is `MatrixFactorizationTrainer` — collaborative filtering; it has **no content-based recommender**. An ML.NET-flavored workaround exists (TF-IDF text featurization feeding a binary classifier trained on purchase labels), but it was rejected for three reasons: (1) the approved topic explicitly promises *weighted interactions* ("završena rezervacija je jači signal od pregleda"), which a trained classifier over text has no natural place for — the deterministic weighted-profile scorer implements the approved description literally; (2) classifier probabilities are not self-explaining — explanations would have to be generated by a parallel content-overlap computation anyway; (3) a deterministic scorer has no training labels, no retraining schedule, no in-memory model lifecycle, and no doc/implementation drift risk. The course mandates "a known algorithm", not ML.NET.

**Final algorithm: content-based filtering — weighted user profile + cosine similarity over categorical feature vectors — blended with popularity, with a pure-popularity fallback for cold-start users.**

## 2. Signals (all written to `UserInteractions` at runtime, all used)

| Signal | InteractionType | Weight |
|---|---|---|
| Booking completed | BookingCompleted | 5 |
| Booking confirmed/paid | BookingConfirmed | 4 |
| Added to favorites | Favorite | 3 |
| Review rated 4–5 | ReviewHigh | 3 |
| Onboarding interest pick | OnboardingInterest | 2 |
| Destination details viewed | View | 1 |
| Search matching a category/tag | Search | 1 |

Recording points (server-side only; the client never posts interactions): details endpoint (View + ViewCount), search endpoint (Search + matched category/tag), favorites service, booking state machine (Confirmed/Completed transitions), review service, onboarding endpoint. Onboarding rows carry `CategoryId`/`TagId` directly and no `DestinationId`. Popularity additionally consumes `ViewCount` and `AverageRating` — both maintained and both used.

## 3. Scoring

**Feature space:** one dimension per DestinationCategory, per Tag, per Region.

**User profile:** aggregate the user's interactions per feature (for destination-linked rows, the features are that destination's category, tags and region; onboarding rows contribute their category/tag directly), summing weights; ×1.5 recency boost for interactions from the last 30 days; normalize to a unit vector.

**Candidates:** every Approved destination the user has no Completed booking for. Destination vector = binary (its category, its tags, its region).
`contentScore = cosine(profile, destinationVector)`
`popularityScore = 0.7·(AverageRating/5) + 0.3·(log(1+ViewCount)/log(1+maxViewCount))`
**Final: `0.8·contentScore + 0.2·popularityScore`**, top N = 10 above a minimum content threshold.

**Cold start:** users with **< 3 total weighted interaction points** get a pure popularity ranking, labeled as such. Onboarding picks (weight 2 each) usually lift new users past the threshold immediately; popularity remains the safety net for skippers.

**Explanations (mandatory), derived from the top dot-product contributors:**
- category → "Recommended because you visited historic fortresses" / "…because you're interested in Nature"
- tag → "Similar to destinations you favorited (tag: waterfalls)"
- popularity/cold start → "Popular right now — highly rated by other travelers"
Each item = `{ destination (light DTO + thumbnail), score, reason }`.

**Second surface (decided KEEP):** destination-details "Similar destinations" — item-to-item cosine over the same feature vectors (no user profile involved, so it works for brand-new users too; it is not the cold-start mechanism — popularity is). Explanation: "Also a medieval fortress in Bosanska Krajina."

## 4. Storage architecture (final: Option A + slim Option C)

**Persisted as input:** only `UserInteractions`. **Computed on demand:** on `GET /recommendations`, the service loads the user's interactions and the Approved destination catalog (catalog cached in `IMemoryCache` — it's hot, rarely-changing data), builds the profile, scores, sorts. Result cached in `IMemoryCache` per user for ~15 min; invalidated on strong interactions (booking, favorite, review, onboarding). **Persisted as output:** every *served* recommendation is appended to `RecommendationLogs` (UserId, DestinationId, Score, Reason, ServedAt) — a pure audit trail, never an input to scoring.

Rejected alternatives, in plain terms:
- **Persisted profile rows** (a table of user→feature weights updated on every interaction): fast reads, but the same information now lives in two places (interactions *and* profile), any missed update path silently corrupts recommendations, and time-decay is awkward because stored weights don't age by themselves. Documented as a scaling path only.
- **Precomputed recommendations** (a background job filling a recommendations table): instant reads, but introduces staleness windows and refresh orchestration for zero benefit at this scale.

### 4a. Defense notes — "why this approach?"

> "Recommendations are computed on demand because at this data scale a full profile build plus scoring takes milliseconds, and on-demand computation guarantees a **single source of truth**: the recommendations always reflect the current interaction data, and the implementation trivially matches the documentation. A **persisted profile** would duplicate the same information in a second representation that can drift out of sync and complicates time-decay weighting; **precomputed recommendations** would add staleness and refresh orchestration without measurable benefit. The only cost of on-demand computation — repeated identical requests — is addressed with `IMemoryCache` (15-minute per-user entries, invalidated on strong interactions), which also demonstrates the course's service-level caching requirement. Finally, `RecommendationLogs` records every recommendation actually shown, with its score and explanation, giving an auditable proof that the explainability requirement is genuinely met and making the recommender's behavior inspectable after the fact — that is our form of 'model management': the model is the documented weight table and formula, versioned together with the code."

## 5. API surface

- `GET /recommendations` → top-N with reasons (JWT user); appends RecommendationLogs rows.
- `GET /destinations/{id}/similar` → top-5 similar.
- `POST /users/onboarding-interests` (list of category/tag ids) → bulk `OnboardingInterest` rows; callable once, skippable.
- Interactions are recorded server-side inside existing endpoints — never posted by the client.
